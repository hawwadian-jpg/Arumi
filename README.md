# Сайт-портфолио Дины Макаровой

Полноценный контентный сайт на **Astro + TypeScript**. Он собирается в быстрые статические HTML-файлы, поэтому не требует администрирования сервера и подходит для простого размещения на Netlify, Cloudflare Pages, Vercel или обычном хостинге.

## Что уже реализовано

- адаптивная главная страница с мобильным меню;
- полноценные русская и английская версии с переключателем языка;
- фиксированный blur-header, который уменьшается при прокрутке и показывает активный раздел;
- отдельные страницы для книг, фильмов, журналистики, путешествий и музыки;
- единый типизированный конфиг для текстов, ссылок и контактов;
- публичный JSON-конфиг по адресу `/api/config.json`;
- собственный минимальный backend формы подписки без сторонних API;
- страницы Privacy Policy, Terms of Use, Thank you и 404;
- SEO-метаданные, Open Graph, sitemap и robots.txt;
- SVG-логотип, favicon и Apple Touch Icon;
- анимации появления с поддержкой настройки «уменьшить движение»;
- проверка TypeScript перед каждой production-сборкой.
- автоматическая GitHub Actions-проверка каждого обновления.

## Быстрый запуск

### Что установить

Нужны:

1. [Node.js 22 LTS](https://nodejs.org/) или новее.
2. `make` — уже установлен на macOS и большинстве Linux-систем. На Windows можно выполнять команды `npm`, указанные ниже.

### Первый запуск

```bash
make install
make dev
```

Откройте [http://localhost:4321](http://localhost:4321). Сервер автоматически обновит страницу после сохранения файла.

Без `make` используются эквивалентные команды:

```bash
npm install
npm run dev
```

## Где менять содержимое

Все основные тексты и ссылки находятся в одном файле:

```text
src/content/site.ts
```

В нём можно изменить:

- заголовок и описание сайта;
- навигацию;
- текст первого экрана;
- блоки Create / Explore / Connect;
- About, цитату и контакты;
- e-mail и Telegram;
- карточки и внутренние страницы направлений.

Русские и английские тексты находятся в объектах `ru` и `en`. Основная русская версия открывается на `/`, английская — на `/en`. Страницы работ и документы также имеют обе локализации.

Изображения хранятся в `public/images`. Если заменить файл, сохранив его имя, менять код не потребуется. Для нового изображения укажите путь вида `/images/example.jpg`.

> В публичный конфиг нельзя добавлять пароли, API-ключи и другие секреты: содержимое доступно любому посетителю по `/api/config.json`.

## Проверка перед публикацией

```bash
make check
make build
make preview
```

- `check` проверяет типы и компоненты;
- `build` создаёт готовый сайт в каталоге `dist`;
- `preview` показывает именно production-сборку.

## Альтернативный деплой: Netlify

Основной production-сценарий проекта — собственный VPS и домен `seeonline.ru` из следующего раздела. Netlify остаётся альтернативой, если отдельный сервер не нужен.

### Вариант 1: через GitHub

1. Загрузите репозиторий на GitHub.
2. Зарегистрируйтесь на [Netlify](https://www.netlify.com/).
3. Нажмите **Add new site → Import an existing project**.
4. Выберите GitHub и этот репозиторий.
5. Netlify сам прочитает `netlify.toml`. Команда сборки будет `npm run build`, каталог публикации — `dist`.
6. В **Site configuration → Environment variables** добавьте:
   - ключ: `SITE_URL`;
   - значение: итоговый адрес, например `https://dina-makarova.netlify.app`.
7. Запустите Deploy. Каждый новый push в основную ветку будет публиковаться автоматически.

### Вариант 2: вручную без GitHub

1. Выполните `make install` и `make build`.
2. Откройте [Netlify Drop](https://app.netlify.com/drop).
3. Перетащите туда каталог `dist`.

Ручной способ публикует сайт, но автоматические обновления и корректный `SITE_URL` удобнее настроить через GitHub.

## Форма подписки

На VPS форма обрабатывается локальным сервисом `server/subscription-server.mjs`. Он доступен только через Nginx-маршрут `/api/subscribe`, проверяет e-mail, honeypot и частоту запросов, а затем сохраняет записи в:

```text
/var/lib/seeonline/subscribers.ndjson
```

Посмотреть подписчиков на сервере можно командой:

```bash
sudo cat /var/lib/seeonline/subscribers.ndjson
```

Сервис запускается через systemd как непривилегированный пользователь `www-data`. При обычном `make deploy` backend также обновляется и безопасно перезапускается.

На Netlify потребуется заменить `/api/subscribe` собственной function или внешним сервисом формы; основной production-сценарий рассчитан на VPS.

## Деплой на собственный сервер: seeonline.ru

Автоматизация рассчитана на чистый Ubuntu/Debian-сервер, уже доступный по DNS как `seeonline.ru`. Скрипты устанавливают Nginx и Certbot, включают HTTPS, создают каталоги релизов и публикуют сайт атомарно.

### 1. Подготовить DNS

У провайдера домена должна существовать A-запись:

```text
seeonline.ru → публичный IPv4-адрес сервера
```

Если используется IPv6, добавьте AAAA-запись. Проверить DNS можно командой:

```bash
dig +short seeonline.ru
```

Адрес в ответе должен совпасть с сервером. Для HTTP/HTTPS извне должны быть доступны TCP-порты 80 и 443.

### 2. Создать локальный deploy-конфиг

```bash
make deploy-config
```

Откройте `deploy.env` и замените `SERVER_HOST`. Файл исключён из Git. По умолчанию используются:

```text
DOMAIN=seeonline.ru
SERVER_USER=root
SSH_PORT=22
TLS_MODE=custom
```

### 3. Настроить SSH-ключ — рекомендуемый способ

Если ключа ещё нет:

```bash
ssh-keygen -t ed25519 -C "seeonline.ru deploy"
ssh-copy-id -p 22 root@АДРЕС_СЕРВЕРА
```

Проверьте вход:

```bash
ssh root@АДРЕС_СЕРВЕРА
```

Перед первым подключением сверьте SSH fingerprint с панелью провайдера:

```bash
make server-key
```

Команда только показывает ключ и не изменяет `~/.ssh/known_hosts`. Если SSH сообщает `REMOTE HOST IDENTIFICATION HAS CHANGED`, не удаляйте старый ключ вслепую: сначала подтвердите новый fingerprint у провайдера сервера.

Скрипты всегда сначала пытаются подключиться по SSH-ключу. Если ключ не работает, они запрашивают пароль и используют `sshpass`, не передавая пароль аргументом командной строки. Для fallback установите `sshpass` отдельно; безопаснее всё же использовать ключ.

### 4. Один раз подготовить сервер

```bash
make server-setup
```

Первичная настройка должна выполняться с `SERVER_USER=root`. Скрипт:

- устанавливает Nginx и системные зависимости;
- устанавливает минимальный Node.js-обработчик подписок и systemd unit;
- создаёт `/var/www/seeonline.ru/releases`;
- устанавливает конфигурацию Nginx;
- проверяет и устанавливает сертификат провайдера из `.secrets`;
- открывает OpenSSH и Nginx Full, если UFW уже активен;
- проверяет конфигурацию перед перезапуском Nginx.

### 5. Опубликовать сайт

```bash
make deploy
```

Перед каждым релизом выполняется Astro/TypeScript-проверка. Затем сборка передаётся по SCP, распаковывается в отдельный каталог и одной операцией переключает симлинк `current`. Незавершённая загрузка не ломает работающую версию. После переключения выполняется HTTPS health-check.

На сервере сохраняются пять последних релизов.

### 6. Вернуть предыдущую версию

```bash
make rollback
```

Rollback переключает `current` на предыдущий релиз, проверяет Nginx и перезагружает конфигурацию.

### Сертификат провайдера

Проект по умолчанию использует сертификат провайдера (`TLS_MODE=custom`). Рекомендуется перевыпустить предоставленный приватный ключ, поскольку ключ, отправленный в сообщении, больше нельзя считать секретным. После перевыпуска:

1. сохраните цепочку как `.secrets/seeonline.ru.fullchain.pem`;
2. сохраните новый ключ как `.secrets/seeonline.ru.key`;
3. выполните `chmod 600 .secrets/seeonline.ru.key`;
4. установите `TLS_MODE=custom` в `deploy.env`;
5. выполните `make cert-check`;
6. выполните `make server-setup`.

`.secrets` полностью исключён из Git. Скрипт сравнит открытые ключи, домен и срок действия до отправки файлов на сервер.

Let’s Encrypt оставлен как запасной режим: установите `TLS_MODE=letsencrypt`, если решите отказаться от сертификата провайдера.

## Частые вопросы

### Почему Astro, а не React или Next.js?

Сайт в основном состоит из контента. Astro отдаёт обычный HTML и подключает JavaScript только там, где он действительно нужен. Это уменьшает размер страницы и количество мест, которые могут сломаться.

### Нужна ли база данных?

Для текущих текстов и страниц — нет. Контент версионируется вместе с кодом и публикуется одной сборкой. База данных понадобится, только если появится вход в админку, частое редактирование без Git или персональные данные пользователей.

### Как добавить ещё одно направление?

Добавьте объект с одинаковым `slug` в массивы `ru.works` и `en.works` внутри `src/content/site.ts`. Astro создаст маршруты `/work/новый-slug` и `/en/work/новый-slug`. Изображение нужно положить в `public/images`.

### Где посмотреть публичный конфиг?

Локально: [http://localhost:4321/api/config.json](http://localhost:4321/api/config.json). После публикации замените домен на адрес сайта.

## Важные замечания перед запуском

- Тексты внутренних страниц сейчас подготовлены как качественные заготовки. Их стоит заменить окончательными биографическими и проектными материалами.
- Юридические страницы являются базовыми шаблонами, а не юридической консультацией.
- Для корректных превью в соцсетях желательно позже добавить отдельное изображение размером 1200×630 px.
- После подключения собственного домена обновите `SITE_URL` в Netlify и запустите новый deploy.
